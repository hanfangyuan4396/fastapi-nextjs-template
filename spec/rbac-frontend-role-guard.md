### 需求描述
- 为前端提供“基于角色的显示/隐藏与路由守卫”能力：
  - 普通用户登录后仅能看到并访问 `students` 页面。
  - 管理员用户登录后可看到并访问 `students-management` 页面（学生管理）。
  - 导航栏中仅对管理员显示“学生管理”按钮；普通用户不可见且无法通过路径直接访问。

### 需求目标
- 根据当前登录用户角色动态渲染导航项与页面内容。
- 对管理员页面提供前端路由守卫：未满足角色要求时禁止访问并进行友好处理（重定向或提示）。
- 方案与现有 JWT + 刷新令牌机制兼容，不破坏透明刷新与登录流程。
- 国际化兼容（沿用现有 i18n 结构）。

### 技术方案
- 角色来源（已决）：在 `access_token` 的 JWT payload 中包含单一 `role`，登录与刷新均返回包含 `role` 的新 `access_token`。前端仅解码 JWT（不验签）用于 UI 控制，安全决策仍由后端 RBAC 强制执行。
- 前端状态：
  - 将当前用户的 `role` 维护在内存（与 `access_token` 同生命周期）。刷新成功时同步更新。
  - 页面初次进入时，若无内存令牌，延续现有 `RequireAuth` 流程触发 `/auth/refresh` 静默续期，成功后可获得包含角色信息的新 `access_token`（或响应体中的 `role`）。
- 导航控制：
  - 导航栏根据 `role` 渲染菜单项：仅 `admin` 显示 `students-management`。
  - 在角色未知的短暂加载态，隐藏敏感项避免闪现。
- 路由守卫：
  - 新增轻量组件 `RequireRole`（或在页面中内联判断）：当当前用户 `role !== 'admin'` 时，统一重定向到首页 `/`（可附带“无权限访问”提示文案）。
  - `students-management` 页面用 `RequireAuth` + `RequireRole('admin')` 组合保护。
- 安全说明：
  - 导航隐藏与前端守卫仅提升 UX，不承担安全职责；真正的权限校验已在后端 RBAC 中强制执行。

### 交互与 UI
- 导航栏：
  - 管理员看到：`首页`、`学生`、`学生管理`。
  - 普通用户看到：`首页`、`学生`。
  - 未确定角色/加载中：不显示 `学生管理`，避免闪烁。
- 页面访问：
  - 普通用户访问 `/students-management`：统一重定向到 `/`。
  - 未登录：沿用 `RequireAuth` 重定向到 `/login?next=…`。
- 国际化：
  - 使用既有 `common.json` 文案键；新增“无权限访问”等提示文案键（如需）。

### 错误与边界处理
- `access_token` 中无 `role` 且响应体未返回 `role`：
  - 触发一次刷新以获取最新信息；仍无则按未知角色处理（隐藏敏感项、拒绝管理员页面）。
- 刷新失败或无 Cookie：
  - 清理本地状态，跳转登录（沿用现有逻辑）。
- 角色快速切换（极少见，后台改权后）：
  - 下一次刷新或重新登录后前端生效；管理员降级为普通用户时，导航立即隐藏敏感项，访问管理员页会被守卫拦截。

### 验收标准
- 普通用户登录：
  - 导航不显示“学生管理”。
  - 直接访问 `/students-management` 被禁止（重定向或 403）。
  - 可正常访问 `/students`。
- 管理员登录：
  - 导航显示“学生管理”。
  - 可访问 `/students-management` 并正常操作。
- 刷新后行为一致：
  - 令牌过期触发刷新后，导航与页面权限与角色一致，无闪现误展示。
- 后端强制校验生效：
  - 前端即使绕过 UI 控制发起管理员接口，也被后端拒绝（403）。
